# PetPal V1.5 体验优化设计方案

**Date**: 2026-05-12  
**Status**: 设计方案，待实现拆分  
**目标读者**: 产品 owner、设计、前端开发

---

## 1. 明确结论

PetPal 当前已经跑通“任务 -> 审核 -> 积分 -> 宠物互动”的基础闭环，但体验层仍偏“功能可用”，距离“让孩子主动想回来照顾宠物、顺手完成当下习惯任务”还有明显差距。

本轮设计优化的核心结论：

1. 顶层信息架构改为 `任务 / 宠物 / 家长` 三个底部 tab，稳定承载三类顶层目标。
2. `宠物` 仍是默认落点，但不再只是展示页，而是“情感锚点 + 当前任务提示页”。
3. `抚摸` 从独立按钮改为对宠物本体的自然触摸交互，减少“按钮感”，强化陪伴感。
4. `任务` 从平铺列表改为“现在该做 / 稍后再做 / 已完成或待审核”的时间驱动任务流。
5. `家长` 从“PIN + 审核 + 开关”升级为“PIN + 快速审核 + 任务配置中心”，支持新增自定义任务。

一句话版本：

> 用宠物承接情绪，用时间段组织任务，用低摩擦审核和自定义任务承接家长真实使用场景。

---

## 2. 设计目标

### 2.1 产品目标

- 让孩子更容易理解“现在该做什么”
- 让孩子更愿意主动打开小程序
- 让家长更容易配置任务并坚持审核
- 让宠物成为习惯提示器，而不是任务列表的装饰

### 2.2 本轮不做

- 不引入云同步
- 不做多孩子 / 多宠物
- 不新增第三方依赖
- 不把系统改成重游戏化或复杂成长系统
- 不引入音效 / 语音能力

## 3. 当前版本的主要问题

结合当前实现和文档，问题主要有四类：

### 3.1 顶层导航不稳定

当前 [app.json](/Users/liumeng/private/mini_program/petpal/app.json) 只有页面路由，没有 `tabBar`。  
当前首页 [pages/index/index.wxml](/Users/liumeng/private/mini_program/petpal/pages/index/index.wxml) 通过“今日任务入口”和“家长入口”跳页，顶层切换依赖按钮，不利于形成稳定心智模型。

### 3.2 宠物交互仍偏按钮驱动

当前首页保留 `喂食 / 玩耍 / 抚摸` 三个并列动作按钮，`抚摸` 仍是显式按钮，不像“在照顾宠物”，更像“点击功能”。

### 3.3 任务列表没有“时间感”

当前任务页 [pages/tasks/tasks.wxml](/Users/liumeng/private/mini_program/petpal/pages/tasks/tasks.wxml) 是按状态平铺，没有“现在该做什么 / 稍后再做什么”的组织方式。  
对 4-7 岁孩子来说，“今天共有 5 个任务”不如“现在先去刷牙”容易理解。

### 3.4 家长页缺少任务配置能力

当前家长页 [pages/parent/parent.wxml](/Users/liumeng/private/mini_program/petpal/pages/parent/parent.wxml) 只支持审核和任务开关，不支持新增、编辑和定时配置任务，无法适配真实家庭的差异化作息。

## 4. 设计原则

### 4.1 顶层导航必须稳定

外部参考中，Apple HIG 强调 tab bar 用于顶层区域切换而不是动作入口；Material 也建议底部导航承载 3-5 个同等重要的顶层目的地，并保持直接可达。

对应到 PetPal：

- `任务 / 宠物 / 家长` 都是顶层目的地，不应该藏在页面内部入口里
- tab 负责切换区域，不负责执行动作
- tab 标签必须短、直白、可预期

### 4.2 习惯任务要靠“日常结构”，不是靠记忆

CDC 与 AAP 面向低龄儿童的建议都强调：规律、可预测、可重复的 routine 能降低对抗、提升理解。  
对应到 PetPal：

- 任务不应只按“是否完成”组织
- 任务应按时间段和当下上下文组织
- 页面文案要告诉孩子“现在去做什么”，而不是只告诉孩子“你还有几项没做”

### 4.3 奖励必须具体且尽量即时

CDC 建议奖励要具体、年龄适配、并在行为后尽快给出。  
对应到 PetPal：

- 孩子完成任务后，反馈不应只是一句“已提交”
- 需要更具体的反馈，如“刷牙完成，小狗在等家长确认”
- 对不需要审核的任务，应即时到账，不再强制等待家长

### 4.4 Prompt 要简单，并且指向“现在就做”

Fogg Behavior Model 认为行为发生需要 `Motivation + Ability + Prompt` 同时到位。  
对应到 PetPal：

- `宠物` 提供 Motivation
- 简单的大按钮和清晰卡片提供 Ability
- “现在先去做刷牙”这类即时提示提供 Prompt

## 5. 新的信息架构

## 5.1 顶层结构

底部 tab 固定为：

1. `任务`
2. `宠物`
3. `家长`

设计说明：

- 左侧 `任务`：孩子的行为入口
- 中间 `宠物`：情感核心页，也是默认首页
- 右侧 `家长`：家长后台，进入后仍需 PIN

### 5.2 默认落点

虽然 tab 顺序按用户要求采用 `任务 / 宠物 / 家长`，但首次设置完成后和日常打开时，默认仍进入 `宠物` tab。

原因：

- 产品的情绪锚点是宠物，不是任务
- 孩子打开后先看到宠物状态，更容易形成“我要去照顾它”的动机
- “现在要做什么”可以通过宠物页的主提示卡承接，不需要把默认页改成任务页

### 5.3 tab 显示规则

- `任务` tab：显示文本标签和图标；有“现在可做任务”时显示红点或数字 badge
- `宠物` tab：作为默认高亮项，视觉可更饱满，但第一版实现优先使用原生样式
- `家长` tab：始终可见；未认证时进入 PIN 验证，不在 tab 上展示待审数量，避免孩子对“审核数字”产生额外关注

### 5.4 实现建议

- 第一阶段优先使用小程序原生 `tabBar`
- 如果后续确实需要中间凸起宠物按钮、动态 badge、或更强的选中态，再评估自定义 tabBar

这样能先用最小改动建立稳定 IA，再决定是否为视觉增强付出额外实现复杂度。

## 6. 页面级设计

## 6.1 宠物页：从展示页升级为“陪伴页 + 当前任务提示页”

### 页面目标

- 孩子一进来就明白宠物现在状态
- 孩子一眼知道“现在优先去做什么”
- 孩子可以自然触摸宠物，而不是主要靠按钮

### 建议结构

1. 顶部天气/时间感区域
   显示早晨、白天、晚上场景变化，让孩子感知时间段
2. 中央宠物主舞台
   放大宠物主视觉，保留情绪、轻动画和说话泡泡
3. 当前任务提示卡
   主文案示例：
   `小狗在等你先去刷牙`
   `现在是睡前时间，先读绘本吧`
4. 宠物照顾区
   保留 `喂食`、`玩耍` 两个显式动作
   `抚摸` 改为直接触摸宠物触发
5. 状态与积分区
   饱食、快乐、积分保留，但视觉层级下沉，不与主提示抢焦点

### 交互重构

#### 抚摸

- 删除独立 `抚摸` 按钮
- 点击宠物头部：触发 `tap_head`
- 点击宠物身体：触发 `tap_body`
- 长按或轻扫宠物身体 `600ms+`：触发 `pet_loop`

反馈建议：

- 轻拍头部：耳朵抖动、小爱心、短泡泡
- 抚摸身体：尾巴摇动、脸部变开心、快乐值缓慢 `+1`
- 如果宠物当前 `sad / sick`，抚摸反馈优先输出安抚文案

这样更符合“摸小狗”的直觉，也能把 `pet-avatar` 已经预留的命中区域能力真正用起来。

### 宠物提示文案机制

宠物说话泡泡不再只由 mood 决定，而是改为：

`宠物状态 + 当前时间段 + 当前任务推荐` 联合决定。

示例：

- 早晨且刷牙未完成：`我准备好啦，先去刷牙吧`
- 晚上且读绘本未完成：`读完故事，我们就准备睡觉`
- 当前没有 urgent 任务：`今天做得不错，来陪我玩一会儿`
- 积分不足且有待做任务：`先完成一个任务，就能来喂我啦`

---

## 6.2 任务页：从“状态列表”改为“时间驱动任务流”

### 页面目标

- 减少孩子一次看到太多任务的压力
- 强化“现在优先做什么”
- 让已完成和待审核状态同样清楚，但不抢主要行动位

### 新结构

任务页分三段：

1. `现在去做`
   当前时间段内推荐优先完成的任务
2. `今天稍后`
   今天会做，但还没到时间窗口的任务
3. `已经完成`
   包含 `pending / approved`，以及当日被驳回可重试项

### 卡片字段

每张任务卡建议展示：

- 图标
- 名称
- 时间标签
- 审核标签：`需要家长确认` / `完成后直接奖励`
- 状态标签：`现在做` / `待确认` / `已完成` / `重新完成`
- 主按钮

### 主按钮规则

- `ready`：`去完成`
- `pending`：禁用，显示 `等家长确认`
- `approved`：禁用，显示 `今天完成`
- `rejected`：可点，显示 `重新完成`
- `upcoming`：禁用，显示 `稍后再做`

### 排序规则

任务页排序不再按原始数组顺序，而是按以下优先级排序：

1. 当前时间窗口内未完成
2. 已过建议时间但仍未完成
3. 稍后时间窗口的任务
4. 待审核
5. 已完成

这样孩子打开任务页时，第一屏总是最该做的事。

---

## 6.3 家长页：从“审核页”升级为“审核 + 配置中心”

### 页面目标

- 家长 30 秒内完成日常审核
- 家长 1-2 分钟内完成任务配置
- 支持新增自定义任务，不逼迫家长接受固定预设

### 新结构

PIN 验证后，家长页分三块：

1. `待审核`
2. `今日任务管理`
3. `任务模板与自定义任务`

### 待审核区

保留并强化：

- `全部通过`
- `全部驳回`
- 单条通过 / 单条驳回

建议增加：

- 仅显示 `今天` 为默认筛选
- 提供 `查看历史待审` 二级入口，避免主屏过载

### 今日任务管理区

展示今天生效的任务清单：

- 是否启用
- 建议时间
- 是否需要审核
- 当前状态汇总

### 自定义任务区

新增入口：`+ 添加任务`

第一版创建流程建议为单页表单或底部弹层，不要做多步 wizard。

## 7. 任务模型重构

## 7.1 设计目标

当前 `utils/state.js` 中的任务结构只有：

- `id`
- `icon`
- `label`
- `points`
- `enabled`
- `requireConfirm`

它足够支撑“固定预设任务 + 开关”，但不足以支撑“时间提示 + 自定义任务 + 审核分流”。

### 7.2 建议任务结构

```js
{
  id: "brushTeeth",
  sourceType: "preset",       // preset | custom
  icon: "🪥",
  label: "刷牙",
  points: 1,
  enabled: true,
  requireConfirm: true,
  schedule: {
    period: "morning",        // morning | noon | afternoon | evening | bedtime | anytime
    startAt: "07:00",
    endAt: "09:00",
    daysOfWeek: [1, 2, 3, 4, 5, 6, 0]
  },
  promptText: "先去刷牙吧",
  sortOrder: 10,
  createdAt: 0,
  updatedAt: 0
}
```

### 7.3 必要字段

本轮至少新增：

- `sourceType`
- `schedule`
- `promptText`
- `sortOrder`

这样才能支持：

- 区分预设任务和自定义任务
- 计算“现在该做什么”
- 让宠物提示文案更自然
- 让任务排序可控

### 7.4 自定义任务字段建议

自定义任务创建表单建议包含：

- 图标
- 名称
- 时间段
- 是否需要审核
- 积分
- 是否启用
- 重复星期

其中前四项是必填核心项，后两项可提供默认值，减少配置负担。

### 7.5 状态模型建议

底层持久化仍保留 `pending / approved / rejected`。  
页面展示层新增派生状态：

- `upcoming`
- `ready`
- `overdue`
- `pending`
- `approved`
- `rejected`

这样可以在不推翻现有 `checkins` 模型的前提下，补足时间驱动 UI。

## 8. 默认任务模板建议

现有预设任务可以保留，但建议补默认时间属性：

| 任务 | 默认时间段 | 说明 |
|-----|-----------|------|
| 刷牙 | morning | 先支持每天 1 次；如果家庭需要早晚各一次，用自定义任务补足 |
| 吃饭 | evening | 当前“吃饭”语义太宽，建议后续鼓励家长拆成“早餐吃完 / 晚饭吃完” |
| 按时睡觉 | bedtime | 睡前核心任务 |
| 读绘本 | bedtime | 和睡前 routine 联动 |
| 整理玩具 | evening | 适合作为晚间收尾任务 |
| 洗澡 | bedtime | 默认关闭，家长按需开启 |

说明：

- 这是一套 V1.5 过渡模板，不是最终家庭作息模型
- 对高频任务，不强行在底层一次性支持“每天多次”，优先通过“自定义任务 + 时间段”解决

这样改动更小，也更易兼容现有存量数据。

## 9. 视觉与文案方向

### 9.1 视觉方向

- 强化时间感：早晨偏浅黄、傍晚偏橙、夜晚偏蓝紫灰
- 强化主舞台：宠物区域更大，任务提示卡更靠近宠物
- 弱化管理感：减少列表式、后台式观感
- 强化孩子可读性：大图标、大圆角、大按钮、少字

### 9.2 文案方向

避免：

- `待打卡`
- `提交成功`
- `状态异常`

改为：

- `现在去做`
- `等家长确认`
- `今天完成啦`
- `我们先去刷牙`

原则是：多说动作，少说系统状态。

## 10. 推荐实现顺序

## Phase A：导航与数据基座

- 接入 `tabBar`
- 确定默认首页仍为 `宠物`
- 给任务模型补 `schedule / promptText / sourceType / sortOrder`
- 增加任务派生状态计算函数

## Phase B：宠物页重构

- 删除 `抚摸` 按钮
- 接入宠物点击 / 长按 / 轻扫交互
- 增加“当前任务提示卡”
- 根据时间段切换提示文案和页面氛围

## Phase C：任务页重构

- 增加 `现在去做 / 今天稍后 / 已完成` 分组
- 增加时间标签和审核标签
- 增加排序和优先级逻辑

## Phase D：家长页与自定义任务

- 新增创建 / 编辑自定义任务
- 支持配置时间、审核方式、积分、开关
- 优化待审核列表和今日任务管理

## Phase E：联调与回归

- 验证老数据兼容
- 验证无时间任务的降级展示
- 验证真机上的 tab、手势与动画

## 11. 风险点与开放问题

### 风险点

1. 任务模型新增 `schedule` 后，需要补 storage 兼容迁移逻辑。
2. `抚摸` 改为手势交互后，需要在真机重新验证命中区域和误触率。
3. `任务` 与 `宠物` 都会展示任务提示，必须明确优先级，避免信息重复。
4. 自定义任务会提高家长配置能力，也会带来更高的数据脏态和排序复杂度。

### 开放问题

1. 是否要在 V1.5 就支持“同一任务每天多次”？
2. `吃饭` 是否应拆成更具体的默认模板？
3. 对 `requireConfirm=false` 的任务，是否仍要在家长页可见历史记录？
4. 家长页是否要加入“本周完成率”卡片，还是继续保持极简？

当前建议：

- 第一轮不做“多次任务”底层能力
- 先保留现有预设任务 ID，避免破坏存量状态
- 历史记录只做轻量查看，不做复杂报表

---

## 12. 本文对后续实现的约束

- 先改信息架构和任务模型，再做页面细节
- 不要先做视觉重绘而跳过任务时间模型
- 不要引入第三方状态管理或 UI 依赖
- 后续实现必须同步更新 [docs/PetPal-Execution-Checklist.md](/Users/liumeng/private/mini_program/petpal/docs/PetPal-Execution-Checklist.md)
