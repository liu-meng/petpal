# PetPal 角色动画与素材规范

**Date**: 2026-05-07  
**Purpose**: 明确 PetPal 最终角色表现方案、素材交付要求和接入边界  
**目标读者**: 产品、视觉设计、动画、前端开发

---

## 一、明确结论

PetPal 当前目标不是“能显示一个宠物”，而是做出一个真正可交互、持续可扩展、能承担核心情感体验的角色。

结合当前产品定位和现有首页结构，推荐方案如下：

- 最终目标方案：`前视角毛绒宠物 + 2D 参数化角色动画 + canvas 渲染`
- 视觉风格优先级：`可爱毛绒感 > 表情感染力 > 轻量动作反馈 > 全身大幅运动`
- 交互优先级：`呼吸 / 眨眼 / 轻微摇摆 / 点击反馈 / 抚摸反馈 / 喂食与玩耍动作`

这意味着本项目不再把“多张静态图切换”视为最终角色方案。

---

## 二、推荐技术路线

### 2.1 主推荐

对 PetPal 来说，优先选择：

- 角色形态：正面或轻微三分之四侧的陪伴型宠物
- 动画方式：Live2D 风格或同类 2D 参数化角色动画
- 小程序接入：使用 `canvas` 作为角色渲染层

原因：

- 这个项目最关键的是“毛绒、软、可爱、会回应”，不是大量复杂跑跳动作。
- 正面陪伴型角色更适合做呼吸、眨眼、耳朵轻摆、脸部反馈、头部轻晃。
- 相比纯骨骼全身动画，这类方案更容易保住原画质感，不容易做成“拼装感很强的纸片狗”。

### 2.2 备选方案

如果后续产品明确要加入更多全身动作，例如：

- 跑向食物
- 拖拽玩具
- 原地跳跃
- 翻滚
- 走位追逐

则可以切换到：

- `2D 骨骼动画 + canvas runtime`

这类方案更像轻游戏角色，但对分层质量、绑定质量、动作设计要求更高。

### 2.3 不建议作为最终方案的路线

以下方案只能用于临时过渡，不建议作为最终角色方案：

- WXML/WXSS 多层静态切图
- GIF 角色
- 视频角色
- 仅靠 4 张情绪图切换
- 无统一分层规范的“边做边补图”

## 三、与当前代码的关系

当前角色接入点已经存在，但只适合原型验证：

- 角色组件：[components/pet-avatar/index.js](/Users/liumeng/private/mini_program/petpal/components/pet-avatar/index.js)
- 渲染模型：[utils/pet-renderer.js](/Users/liumeng/private/mini_program/petpal/utils/pet-renderer.js)
- 首页交互入口：[pages/index/index.wxml](/Users/liumeng/private/mini_program/petpal/pages/index/index.wxml)

当前实现本质上是：

- 根据 `mood` 和 `action` 选择资源
- 显示立绘
- 播放轻量 UI 动效

后续应演进为：

- `pet-renderer` 负责从状态机输出“动作状态”和“表情参数”
- `pet-avatar` 负责承载角色渲染、命中区域、动画播放和回调
- 首页继续只关心业务动作，如 `feed`、`play`、`pet`

也就是说，后续应替换角色内部渲染方式，而不是重写整个首页。

## 四、角色表现目标

最终角色至少应具备以下感知能力：

- 待机时持续呼吸，不是完全静止
- 低频自动眨眼，节奏不机械
- 点击头部、身体时有不同反馈
- 抚摸时有连贯回应，而不是只闪一下特效
- 喂食、玩耍时有明确进入动作和退出动作
- 开心、普通、委屈、生病四类状态能一眼看出差异

最终角色至少应具备以下技术能力：

- 支持循环待机动作
- 支持一次性动作播放
- 支持动作完成后回到 idle
- 支持热区点击
- 支持被业务状态驱动切换情绪
- 支持低端机降级

## 五、建议的状态与动作清单

### 5.1 基础情绪状态

- `excited`
- `normal`
- `sad`
- `sick`

### 5.2 必做待机动作

- `idle_normal`
- `idle_excited`
- `idle_sad`
- `idle_sick`

这些待机动作应至少包含：

- 呼吸
- 眨眼
- 轻微头部或身体摆动
- 耳朵或尾巴的二级运动

### 5.3 必做交互动作

- `tap_head`
- `tap_body`
- `pet_loop`
- `feed_start`
- `feed_loop`
- `feed_end`
- `play_start`
- `play_loop`
- `play_end`
- `recover`

### 5.4 建议特效动作

- `heart_pop`
- `sparkle_pop`
- `dizzy_loop`
- `heal_pop`

特效应作为附加层，不应替代角色本体动画。

## 六、素材交付要求

### 6.1 主源文件要求

- 源文件格式：`PSD` 或 `PSB`
- 建议画布：`2048 x 2048`
- 背景：透明
- 色彩：统一使用同一套最终调色，不允许同角色不同状态偏色严重
- 构图：统一角色比例、统一镜头、统一透视

所有状态都必须基于同一版角色定稿，不允许每张情绪图重新画一只“长得不太一样的狗”。

### 6.2 分层要求

如果走参数化角色动画，至少需要拆出这些层：

- `body`
- `head`
- `ear_l`
- `ear_r`
- `eye_l`
- `eye_r`
- `eyelid_l`
- `eyelid_r`
- `pupil_l`
- `pupil_r`
- `brow_l`
- `brow_r`
- `mouth_base`
- `mouth_smile`
- `mouth_open`
- `mouth_sad`
- `nose`
- `cheek_l`
- `cheek_r`
- `arm_l_front`
- `arm_r_front`
- `tail`
- `accessory`
- `shadow`

如果有毛发边缘、腮红、耳朵内侧、嘴型变化，也应单独拆层，不要烘焙进主脸。

### 6.3 导出规则

- 所有部件保持统一坐标系
- 不要按单层内容自动裁边到最小包围盒
- 透明区域要保留
- 导出时保持同尺寸对齐
- 部件命名固定，不使用中文、空格、最终版、最新版这类命名

推荐命名格式：

- `dog_body.png`
- `dog_head.png`
- `dog_ear_l.png`
- `dog_ear_r.png`
- `dog_eye_l.png`
- `dog_eyelid_l.png`
- `dog_mouth_smile.png`

### 6.4 特效素材

特效建议单独交付：

- `fx_heart`
- `fx_sparkle`
- `fx_heal`
- `fx_dizzy`

特效可以使用：

- 透明 `PNG`
- 序列帧
- 小图集 + `JSON` 描述

## 七、交互定义要求

为了让前端真正做成交互角色，除素材外还必须提供一份动作配置表。

建议字段如下：

| 字段 | 说明 |
|------|------|
| `state` | 所属情绪状态 |
| `action` | 动作名 |
| `loop` | 是否循环 |
| `duration` | 动作时长 |
| `interruptible` | 是否可打断 |
| `returnTo` | 播放完成后回到哪个 idle |
| `hitArea` | 触发热区 |
| `fx` | 对应特效 |
| `sound` | 对应音效，可选 |

至少需要定义这些热区：

- `head`
- `body`
- `tail`

如果角色带有配饰且可交互，再增加：

- `accessory`

## 八、小程序接入边界

### 8.1 渲染建议

角色本体建议使用：

- `canvas`

不建议把角色主体继续拆成大量普通 `image/view` 叠层。

### 8.2 逻辑分层建议

建议后续代码职责保持如下边界：

- `pages/index/`：业务状态、按钮点击、积分消耗
- `components/pet-avatar/`：角色显示、动作播放、命中反馈
- `utils/pet-renderer.js`：状态到角色动作的映射

### 8.3 性能约束

需要从一开始就控制：

- 首屏角色资源体积
- 单张纹理尺寸
- 同屏特效数量
- 低端机帧率

建议目标：

- 低端机可稳定 `30 FPS`
- 中高端机目标 `60 FPS`
- 首屏角色主资源尽量控制在可接受的小程序包体内

## 十、验收标准

角色方案只有同时满足以下条件，才算可进入正式开发：

- 角色定稿统一，四种情绪没有明显画风漂移
- 分层文件可直接用于绑定，而不是只能看不能动
- 每个动作都有明确命名和回退目标
- 热区定义完整
- 前端知道该接什么数据，不需要靠猜

角色接入完成后，至少应达到以下效果：

- 首页角色不是静态贴图
- 点击头和身体反馈不同
- 抚摸动作可连续响应
- 喂食和玩耍动作有开始、进行、结束
- 情绪切换时角色变化明显且仍保持可爱

---

## 十一、下一步执行顺序

建议严格按以下顺序推进：

1. 先确认最终路线：`参数化陪伴型角色` 还是 `全身骨骼型角色`
2. 先做一版统一角色定稿，不要先切图
3. 按本规范拆分图层并交付主源文件
4. 动画师完成基础待机和交互动作
5. 前端再改造 `pet-avatar` 和 `pet-renderer`

如果第 1 步和第 2 步没完成，不建议继续投入角色程序实现。
